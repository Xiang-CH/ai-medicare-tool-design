# 医生端 · 记录流程（按面板 / 阶段）

## 设计意图

- **首屏不展示中央文书主区**，降低未开始记录时的视觉与认知负担；核心动作收敛为「开始记录」。
- **先采集、后成稿**：全屏对话/录音识别专注倾听与转写；结束后再进入全屏编辑，避免边听边写分散注意力。

---

## 阶段 0：待命（无中央文书主区）

**可见内容**

- **顶部全局条**（患者、就诊类型、同步状态等，按产品需要裁剪）。
- **患者时间线面板**：跨次就诊事件轴，可锚定至原始文书/报告。
- **病史摘要面板**：智能摘要卡片（手术史、过敏、长期用药等）及来源、待核实标记。
- **检验与趋势面板**：关键指标趋势与「今日 vs 历史」对照（可与时间线/摘要分栏或上下分屏，按视口定稿）。
- **不展示中央文书编辑主区**（无 SOAP/病程大块编辑区）；三块信息面板占据主工作区，供接诊前速览。
- **漂浮按钮「开始记录」**：固定在安全区内（如右下角），始终可点，符合单手/快速触达。

**布局提示**

- **左宽右分**：左为时间线；右上摘要、右下检验趋势。

**交互**

- 点击「开始记录」→ 进入阶段 1。

---

## 阶段 1：全屏 · 对话记录识别

**布局**

- **全屏覆盖**当前工作区（含原侧栏的暂时隐藏或压暗），仅保留与采集相关的控件。
- 主区域：**实时转写/对话流**（可显示波形、计时、分段文本）；明确区分「系统听到的」与「已确认片段」（若有多步确认策略）。

**医生操作**

- **结束记录**（显式按钮，如「结束并生成草稿」）：触发转写定稿与结构化抽取，进入阶段 2。
- **取消 / 暂停**（可选）：取消则回到阶段 0；暂停保留草稿但不进入编辑。

**Track A 对齐（简述）**

- 展示**采集状态**（录音中、处理中、失败）与**能力边界**短文案（例如：转写可能出错，终稿以编辑区为准）。
- 失败时：**_alert + 重试 / 跳过转写直接进入空编辑**。

---

## 阶段 2：全屏 · 文书编辑面板（SOAP + 逐句审核）

**界面策略（单一 Editor fullscreen）**

- 文书编辑阶段只有 **一个全屏界面**：**Editor fullscreen**（SOAP 主区 + 逐句审核）。**不**为 diff **另开**独立页面、路由或第二套文书布局。
- **Diff** 是该界面内、针对**已修订句子**的**显示状态**（例如行内删除线/新增底色、或点击展开对照），与「当前句高亮 + 其余灰度」等审核态**同屏切换**，而非跳转。
- **修订漂浮窗**为叠在该全屏上的**模态层**，关闭后仍停留在同一 Editor fullscreen，仅底层正文进入或保持相应 **diff 状态**。

**布局**

- **全屏文书编辑**：中央主区固定展示 **SOAP 四个区块**（Subjective / Objective / Assessment / Plan），每区有标题与正文；内容来自阶段 1 结构化初稿，可辅以侧栏校验、趋势等（不抢占主阅读动线）。
- **分区审核顺序**：按 **S → O → A → P**（或产品自定顺序）推进；**每个 SOAP 区块须单独完成审核**后，才视为该区「已核对」，可进入下一区或总提交（具体是否强制顺序可在交互细则里定稿）。

**逐句审核（核心交互）**

- 在某一 SOAP 区内，审核粒度为**一句话（一句一决策）**；系统每次只聚焦**当前待审的那一句**。
- **视觉**
  - **当前句**：正常对比度、主色高亮（或背景强调），表示「正在裁决」。
  - **其余全文**（含其他 SOAP 区及本区其他句）：**灰度 / 降对比**，可读但明显退后，避免抢注意力。
- **当前句右侧控件**（同一行或紧随其后的行尾对齐）
  - **勾（采纳）**：确认该句内容保留进终稿；进入下一句待审。
  - **叉（否决）**：拒绝该句 → 弹出 **「修订漂浮窗」**（见下节）；不在主界面直接清空，避免误操作且保留对照基准。

**否决后 · 修订漂浮窗（人机协作）**

- **触发**：点击 **叉** 后出现，悬浮于文书主区之上（模态或带遮罩，防止误触背后 SOAP）；关闭前暂停该句的逐句灰度审核流。
- **布局**
  - **左侧（大框）**：展示**当前被否决的那一句**的完整文本（即 AI 初稿该句），**可直接编辑**；为主要工作区，字号与行高便于改写。
  - **右侧（较窄栏）**：**AI 再次建议**区——可展示一条或多条候选句式、「重新生成」按钮、简短说明（置信度/依据提示等按 Track A 节制呈现）；医生可 **一键将某条建议填入左侧**（或插入光标处），再在左侧微调。
- **医生路径**
  - **纯人工**：仅编辑左侧大框，不采用右侧建议。
  - **人机混合**：从右侧选建议 → 填入左侧 → 再手动修改。
- **确认 / 取消**
  - **确认修订**：将左侧最终文本写回该句在 SOAP 中的位置，关闭漂浮窗；**仍停留在同一 Editor fullscreen**，底层该句进入 **diff 显示状态**（见下节）；随后进入下一句审核或再点勾确认本句定稿（二选一：**建议**「确认修订」后自动视为本句通过，或仍需再点一次勾——原型可试后者更符合二次确认精神）。
  - **取消**：关闭窗，该句仍维持否决前状态或回到待审（产品定一种，需与审计一致）。

**同屏 · Diff 状态（仍在 Editor fullscreen）**

- 漂浮窗确认关闭后，**不离开**全屏编辑器；在 **同一 SOAP 主区**内，对该句（或该段）进入 **diff 状态**：
  - **对照基准**：AI 原始生成内容 vs **医生最终写入内容**（增删改用常规 diff 样式：删除线/底色、新增高亮等）。
  - **范围**：至少覆盖本句；若一句内多次往返修改，可折叠为「相对 AI 初稿的最终差异」一条视图。
  - **与审核态共存**：继续审核其他句时，已修订句可保持 **diff 态**或「已改」角标；**展开/收起** diff 细节，避免全文噪声（例如默认仅角标，点击该句展开行内对照）。
- **目的**：满足 **可感知的人工介入**（Track A：透明、可追溯），与审计日志字段对齐。

**Track A 对齐**

- 每一句的采纳/否决、漂浮窗内编辑前后文本、右侧 AI 建议版本可记入**审计日志**（时间、原句、操作），与「差异高亮 / 版本」设计衔接。
- 低置信句可在高亮旁加**次要提示**（如「请核对」），但不替代医生点勾/叉的显式选择。

**其他医生操作**

- 保存、提交、返回：若「返回」到阶段 0，需明确**草稿已保存**、未完成审核的进度是否保留。

---

## 状态流转（摘要）

```text
阶段 0 待命（时间线 + 病史摘要 + 检验趋势；无中央文书主区；漂浮「开始记录」）
    → 点击开始记录
阶段 1 全屏对话记录识别
    → 点击结束
阶段 2 同一 Editor fullscreen（SOAP；逐句高亮 + 灰度 + 勾/叉）
    → 点叉 → 修订漂浮窗（叠层）→ 确认后同屏进入该句 diff 状态
    → 保存后回 0 或停留 2（按信息架构定）
```

## 原型 / Figma 建议画板

| 画板 ID        | 说明           |
|----------------|----------------|
| `Doc_00_Idle`  | 时间线 + 病史摘要 + 检验趋势；无中央文书；漂浮 CTA |
| `Doc_01_Capture_Fullscreen` | 全屏识别 |
| `Doc_02_Editor_Fullscreen`  | **唯一文书全屏**；用 **Variants** 区分：`审核态`（单句高亮 + 其余灰度 + 勾/叉）、`句级 diff 态`（AI 原稿 vs 终稿；可展开/收起）；**否决修订**为同帧上的 **Modal 叠加**（左大框编辑 + 右 AI 再建议） |

文书阶段 **不再单建** `Doc_02_Diff_*` 独立画板；diff 仅 `Doc_02_Editor_Fullscreen` 的变体或同界面状态。
